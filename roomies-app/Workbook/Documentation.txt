SESSION LOG — 2026-06-01 (Changes.md — SSO + signup fix)
==========================================================

BUGS FIXED
----------
1. LyfeWare SSO login loop (persistent) — root cause identified and fixed.
   With detectSessionInUrl:true (Supabase default), the library auto-detected
   the hash tokens AND our code also called setSession() manually.  Supabase v2
   rotates the refresh token on each use; the second call used an already-rotated
   token, Supabase fired SIGNED_OUT, and the user landed back on the login screen.
   Fix A — src/lib/supabase.ts: added detectSessionInUrl:false so only our
   explicit setSession() call ever consumes the tokens.
   Fix B — src/context/AuthContext.tsx: extract tokens and clear the URL hash
   BEFORE subscribing to onAuthStateChange (preventing any race); removed the
   5-second timeout during SSO (it could fire loading=false with null session on
   slow networks); setSession() failure now explicitly resets all auth state so
   the login screen shows immediately rather than hanging.

2. "Database error saving new user" (persistent) — root cause was that the
   lyfeware_unified_schema.sql trigger had no exception block.  Any username
   conflict or DB error caused the trigger to throw, rolling back the entire
   auth.users INSERT and returning HTTP 500.
   Fix — lyfeware_unified_schema.sql: wrapped the entire INSERT in
   BEGIN...EXCEPTION WHEN OTHERS THEN NULL; END; and added base_name sanitization.
   The quick-fix migration is still in supabase/migrations/002_fix_trigger.sql.
   ACTION REQUIRED: run 002_fix_trigger.sql (or the updated schema trigger) in
   the Supabase SQL Editor to patch the live database.

SESSION LOG — 2026-05-31
========================

BUGS FIXED
----------
1. Signup gives "Database error saving new user" (500 on /auth/v1/signup)
   Root cause: supabase/migrations/001_init.sql trigger had no exception handler.
   Any username conflict (e.g. email prefix already taken) caused the profile
   INSERT to fail, propagating the error up through the trigger, rolling back
   the entire auth.users row, and returning HTTP 500.
   Fix: Created supabase/migrations/002_fix_trigger.sql — replaces the function
   with an exception-wrapped version that strips invalid chars from the base
   username, deduplicates with numeric suffixes, and swallows all errors so the
   auth.users INSERT always succeeds. Profile is created client-side on first
   SIGNED_IN if the trigger was skipped.
   IMPORTANT: Run 002_fix_trigger.sql in the Supabase SQL Editor to apply.

2. LyfeWare SSO redirects back to login screen instead of signing in
   Root cause: Race condition in AuthContext.tsx. onAuthStateChange fires
   INITIAL_SESSION (no session yet) before supabase.auth.setSession() resolves,
   calling setLoading(false) with no user → app renders unauthenticated →
   <Navigate to="/welcome"> fires. The subsequent SIGNED_IN event fires after
   the redirect has already happened.
   Fix: Detect the hash access_token BEFORE subscribing to onAuthStateChange.
   The pendingSSO flag skips the loading=false call for INITIAL_SESSION when
   tokens are about to be injected, keeping the Spinner up until SIGNED_IN
   fires and the profile is loaded.
   File changed: src/context/AuthContext.tsx
